# 관리자 페이지 > 보안 리포트 > 해킹탐지 자동스케줄러(어드민 계정 문자발송 포함) 구현 계획

## 1) 목표 및 범위
- 관리자 페이지의 **보안 리포트** 섹션에서 해킹 의심 이벤트를 자동 수집/분류/알림하는 스케줄러를 구축한다.
- 해킹탐지 결과가 임계치를 넘는 경우, 지정된 **어드민 계정에 SMS 자동 발송**을 수행한다.
- 본 문서는 구현 계획 문서이며, 실제 코드 반영 전 아키텍처/운영 정책 합의를 목적으로 한다.

## 2) 요구사항 정의

### 기능 요구사항
1. 주기 스캔
   - 기본 주기: 5분(운영에서 환경변수로 조정 가능)
   - 수집 대상: 로그인 실패, 관리자 API 비정상 호출, 권한 상승 시도, Rate Limit 초과, 의심 IP 패턴
2. 이상 징후 점수화(위험도 스코어)
   - 이벤트 타입별 가중치 부여
   - 동일 IP/계정/디바이스 기반 누적 스코어 계산
3. 리포트 생성
   - 최근 24시간/7일 보안 이벤트 요약
   - 고위험 이벤트 TOP N 및 추세 차트 데이터 제공
4. SMS 알림
   - 조건: 위험도 기준 초과(예: score >= 80) 또는 Critical 룰 매칭
   - 수신자: 활성화된 어드민 계정(전화번호 인증 완료)
   - 중복 방지: 동일 사건군에 대해 쿨다운(예: 30분)
5. 관리자 UI
   - 최신 탐지 실행 시각, 탐지 건수, 위험도 분포
   - 알림 발송 이력(성공/실패, 재시도 횟수)
   - 탐지 룰 ON/OFF, 임계치 수정(권한 기반)

### 비기능 요구사항
- 가용성: 스케줄러 실패 시 재시도/자동복구
- 신뢰성: 같은 이벤트의 중복 처리 방지(idempotency key)
- 보안: 개인정보(전화번호) 암호화/마스킹 저장
- 감사추적: 누가 임계치/룰을 바꿨는지 변경 이력 기록
- 확장성: 룰 추가 시 코드 변경 최소화(룰 설정형 구조)

## 3) 아키텍처 제안

### 구성 요소
1. **Event Collector**
   - 인증/인가/방화벽/애플리케이션 로그를 수집
2. **Detection Engine**
   - 룰 기반 탐지 + 점수화
3. **Scheduler Worker**
   - 주기 실행, 작업 잠금(distributed lock), 재시도
4. **Alert Notifier (SMS Gateway Adapter)**
   - SMS 사업자 API 연동(예: 국내 문자 API)
5. **Security Report API**
   - 관리자 페이지용 집계 조회 API
6. **Admin UI (Security Report)**
   - 보고서 조회, 필터링, 정책 관리

### 데이터 저장소(예시)
- `security_events`: 원본 이벤트
- `security_findings`: 탐지 결과/위험도/사건군 키
- `security_alerts`: 문자 발송 요청/결과/재시도 로그
- `security_rules`: 룰 설정, 임계치, 활성화 상태
- `admin_contacts`: 어드민 연락처(암호화), 알림 수신 여부
- `audit_logs`: 정책 변경 이력

## 4) 탐지 룰(초안)
1. 로그인 실패 폭증
   - 5분 내 동일 IP 실패 N회 이상
2. 관리자 엔드포인트 스캐닝
   - 존재하지 않는 admin 경로 다수 호출
3. 권한 토큰 이상
   - 만료/위변조 의심 토큰 반복 사용
4. 비정상 지역/ASN 접근
   - 짧은 시간에 국가/ASN 급격 변경
5. 비정상 요청 패턴
   - User-Agent 고정 + 고빈도 요청 + 오류율 급증

> 룰별로 `severity`, `score`, `window`, `threshold`, `cooldown`을 파라미터화하여 운영에서 조정 가능하게 설계.

## 5) 스케줄러 실행 플로우
1. 스케줄러 트리거(예: Cron)
2. Distributed lock 획득(중복 실행 방지)
3. 최근 구간 로그 수집
4. 룰 엔진 평가 및 사건군(cluster) 생성
5. 위험도 계산 및 finding 저장
6. SMS 발송 조건 충족 여부 판단
7. 수신자 목록 조회(활성 어드민)
8. SMS 발송 및 결과 저장(실패 시 backoff 재시도)
9. 실행 메트릭 기록 및 lock 해제

## 6) SMS 발송 정책
- Critical: 즉시 발송
- High: 30분 내 누적 기준 도달 시 발송
- Medium 이하: 관리자 화면 리포트에만 표시
- 야간/휴일 정책: 필요 시 보조 수신자 그룹 추가
- 장애 대응: 문자 게이트웨이 실패 시 대체 채널(이메일/푸시) 옵션 준비

## 7) 운영 및 모니터링
- 모니터링 지표
  - 스케줄러 성공률, 평균 실행 시간, 탐지 건수, 오탐률
  - SMS 성공률, 지연 시간, 재시도 비율
- 경보
  - 스케줄러 연속 실패 횟수 초과
  - SMS 발송 실패율 급증
- 대시보드
  - 일/주 단위 트렌드 및 이벤트 유형별 분포

## 8) 보안/컴플라이언스 체크포인트
- 전화번호 등 PII 암호화 저장 및 전송 구간 TLS 강제
- 최소 권한 원칙(알림 설정 수정 권한 분리)
- 로그 보존 주기 및 파기 정책 수립(예: 90일)
- 관리 기능 접근 시 MFA 권장

## 9) 단계별 구현 일정(안)

### Phase 1. 설계/정의 (2~3일)
- 이벤트 스키마, 룰 스키마, 알림 정책 확정
- 관리자 UI 와이어프레임 확정

### Phase 2. 백엔드 구현 (4~6일)
- 수집 파이프라인 + 탐지 엔진 + 스케줄러 + 잠금
- findings/alerts 저장 및 조회 API 구현

### Phase 3. SMS 연동 (2~3일)
- SMS 게이트웨이 어댑터 구현
- 템플릿/재시도/중복방지/쿨다운 적용

### Phase 4. 관리자 UI 구현 (3~4일)
- 보안 리포트 화면, 필터, 상세 패널
- 알림 이력 및 정책 편집 UI

### Phase 5. 테스트/튜닝 (3~5일)
- 부하 테스트, 오탐/미탐 튜닝
- 장애 복구/재시도/락 경합 시나리오 점검

## 10) 테스트 전략
- 단위 테스트: 룰 평가, 점수화, 쿨다운 판단
- 통합 테스트: 스케줄러→탐지→발송 end-to-end
- 보안 테스트: 권한 없는 정책 수정/조회 차단
- 운영 리허설: 모의 공격 로그 주입 후 알림 검증

## 11) 리스크 및 대응
- 오탐 과다: 룰 가중치 조정, 샘플링 리뷰 프로세스 도입
- 문자 비용 증가: severity 기반 발송 차등 + 일일 상한
- 외부 문자 API 장애: 재시도 + 대체 채널 + 장애 알람
- 로그 품질 편차: 수집 전처리/정규화 단계 강화

## 12) 산출물
- 보안 리포트 기능 명세서
- 탐지 룰 카탈로그(v1)
- 운영 런북(장애 대응 절차)
- SMS 템플릿 및 발송 정책 문서
