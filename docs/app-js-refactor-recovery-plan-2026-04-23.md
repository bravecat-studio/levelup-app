# app.js 리팩토링 이후 기능 구현 불가 이슈 — 원인 가설 및 해결방안 (2026-04-23)

## 1) 문제 정의
- 현상: app.js 리팩토링 이후 신규 기능 구현/검증이 사실상 불가능한 상태.
- 특징: 원인이 단일 에러라기보다 **초기 부팅 체인 실패 + 전역 브리지 결합도** 문제일 가능성이 높음.

---

## 2) 현재 코드 기준 핵심 리스크 포인트

### A. 부팅 초기에 단일 예외로 전체 흐름이 멈출 수 있음
`DOMContentLoaded`에서 `bindEvents()`가 대량 DOM id를 전제하고 직접 `addEventListener`를 호출합니다.
- 특정 id가 하나라도 누락되면 `null.addEventListener`로 즉시 중단
- 이후 초기화(권한, 소셜, 플래너, 탭 로직)까지 연쇄적으로 미실행

### B. Window Bridge 결합도가 높아 리팩토링 시 회귀 폭이 큼
도메인 모듈들이 `window.*` 함수/상태를 참조하는 구조라,
- 함수명 변경
- 노출 타이밍 변경
- 모듈 로딩 순서 변화
중 하나만 발생해도 런타임에서 늦게 터집니다.

### C. 탭 전환/광고/모달 등 횡단 관심사(cross-cutting) 의존이 많음
`switchTab`, `AdManager`, `SocialModule`, `renderReelsFeed` 등 서로 다른 도메인이 교차 호출되어
부분 수정 시 영향 범위가 예상보다 크게 퍼집니다.

### D. “정적 검사 없이 런타임 확인” 패턴
현재는 빌드 단계에서 브리지 누락/DOM 누락을 차단하는 게이트가 약합니다.
즉, 문제를 배포 직전/배포 후에 발견하게 됩니다.

## 3) 해결 전략 (권장: 3단계)

## Phase 1 — 오늘 바로 가능한 긴급 복구
1. **부팅 생존성 확보 (Crash-stop 제거)**
   - `bindEvents`를 “필수/선택” 바인딩으로 분리
   - 선택 요소는 없으면 경고만 남기고 진행
2. **부팅 실패 가시화**
   - `window.__BOOT_HEALTH__`에 실패 지점(함수명/DOM id/스택) 기록
   - 관리자 로그 패널에서 즉시 확인 가능하도록 노출
3. **기능 플래그로 분리 배포**
   - 신규 리팩토링 모듈은 feature flag ON일 때만 활성화
   - OFF 시 기존 경로로 즉시 폴백 가능하게 유지

### Phase 1 완료 기준
- 앱이 “일부 기능 저하” 상태여도 로그인/탭 전환/핵심 저장은 동작
- 단일 null 참조로 전체 부팅이 멈추지 않음

---

## Phase 2 — 원인 고립 (1~2일)
1. **브리지 계약(Contract) 명세 파일 도입**
   - 예: `docs/contracts/window-bridge-contract.md`
   - `producer(app.js/모듈)` vs `consumer(도메인 모듈)`를 키 단위로 표준화
2. **자동 점검 스크립트 추가**
   - `window.X` 소비 키와 실제 노출 키를 비교하는 정적 검사
   - CI에서 불일치 시 fail
3. **DOM 바인딩 감사 스크립트 추가**
   - `bindEvents`에서 참조하는 id 목록 vs `app.html` 실제 id diff 출력

### Phase 2 완료 기준
- “이름 변경/순서 변경”으로 인한 회귀를 PR 단계에서 차단
- 원인 불명 상태 해소(누락 항목이 리포트로 즉시 식별)

---

## Phase 3 — 구조 안정화 (3~7일)
1. **app.js를 오케스트레이터로 축소**
   - 도메인 로직 직접 구현 금지, 호출/조립만 담당
2. **의존성 주입(Dependency Injection) 일원화**
   - 도메인 모듈에서 `window` 직접 참조 최소화
   - 생성자 인자로 주입받은 의존성만 사용
3. **탭 단위 회귀 스모크 테스트 고정**
   - status/social/reels/dungeon/planner 최소 시나리오 자동화

### Phase 3 완료 기준
- 신규 기능 구현 시 “기존 탭 전체 재검증” 부담 감소
- 수정 범위가 도메인 내부로 제한됨

## 4) 바로 적용 가능한 운영 가이드 (실무용)

## 브랜치 운영
- `hotfix/boot-survivability` : null-safe 이벤트 바인딩, 부팅 헬스 로그
- `chore/bridge-contract-check` : 브리지 계약 검사 스크립트
- `refactor/domain-isolation-*` : 도메인별 분리(기능 플래그 필수)

## 배포 게이트
- Gate 1: 앱 부팅 60초 내 console error 0건(치명 에러 기준)
- Gate 2: 탭 전환 6개 핵심 플로우 통과
- Gate 3: 로그인/저장/동기화/광고 4개 핵심 이벤트 성공

## 롤백 규칙
- 신규 모듈에서 치명 오류 발생 시 feature flag OFF로 즉시 복구
- 같은 배포에서 구조개편 + 기능추가 동시 진행 금지

---

## 5) 우선순위 제안 (현재 상황 최적화)
1. **오늘:** 부팅 중단 제거 + 로그 가시화
2. **내일:** 브리지/DOM 정적 검사 스크립트 CI 연결
3. **이번 주:** app.js 직접 구현 축소(도메인 주입 방식 통일)

---

## 6) 기대 효과
- “원인 불명” 상태를 **측정 가능한 누락 리포트 상태**로 전환
- 기능 구현 불가 → 부분 기능 저하 + 빠른 핫픽스 가능한 상태로 전환
- 리팩토링 속도보다 **복구 가능성(operability)**을 먼저 확보
